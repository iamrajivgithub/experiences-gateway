Tab 5

Overlap Resolutions

Six design tensions identified during brainstorm — and the agreed resolution for each.

1 — Availability endpoint ownership

Availability data lives in PostgreSQL (booking-service domain) but is needed when viewing an experience detail page.

graph LR FE["Frontend"] -->|"GET /api/v1/experiences/abc"| ES["experience-service\n:8002\n(MongoDB)"] FE -->|"GET /api/v1/experiences/abc/availability"| BK["booking-service\n:8003\n(PostgreSQL)"] ES -->|"200 { title, images, price... }"| FE BK -->|"200 { available_dates: [...] }"| FE style BK fill:#1A3A5C,color:white style ES fill:#1A3A5C,color:white
❌ Rejected approach
experience-service internally calls booking-service to merge data. Creates runtime coupling — if booking-service is down, experience detail pages break.
✅ Agreed resolution
Two separate calls from the frontend. booking-service owns GET /api/v1/experiences/{id}/availability. Clean boundary, independent failure modes.

2 — Rating data on experience cards

Experience list needs rating + review_count per card. Reviews live in review-service (PostgreSQL), experiences in experience-service (MongoDB).

sequenceDiagram participant Admin participant RV as review-service participant PG as PostgreSQL reviews participant MG as MongoDB experiences Admin->>RV: PATCH /api/v1/reviews/{id}/approve RV->>PG: UPDATE reviews SET approved=true RV->>MG: UPDATE experiences SET rating_avg=4.97, review_count=143 Note over MG: rating_avg and review_count
stored directly on experience doc RV-->>Admin: 200 OK Note over MG: experience-service reads these
fields on GET — no extra call needed
❌ Rejected approach
experience-service calls review-service on every list request to aggregate ratings. N+1 problem — 12 cards = 12 calls to review-service.
✅ Agreed resolution
review-service writes rating_avg and review_count back to MongoDB when a review is approved. experience-service reads them as plain fields. Zero inter-service calls at read time.

3 — Wishlist enrichment

Wishlist is a list of experience IDs stored in PostgreSQL (user-service). Frontend needs titles and images to display the list.

sequenceDiagram participant FE as Frontend participant US as user-service :8004 participant ES as experience-service :8002 FE->>US: GET /api/v1/users/me/wishlist US-->>FE: 200 { items: [ { id: "abc", title: "Villa Paradiso", image_url: "...", price: 4200 } ] } Note over US: Returns cached snapshot
stored in wishlist_cache JSONB column FE->>ES: GET /api/v1/experiences/abc (optional — for full detail) ES-->>FE: 200 { full experience object }
❌ Rejected approach
user-service calls experience-service internally to enrich wishlist items. Tight coupling, slower response.
✅ Agreed resolution
user-service stores a wishlist_cache JSONB field (title + image_url + price) written at save time. Returns snapshot instantly. Frontend calls experience-service only when user opens full detail.

4 — Search vs experience-service

Both services query the MongoDB experiences collection. Is search-service redundant?

experience-service scope
Structured browse — category tabs, known filters (city, price range), pagination. Simple find() queries on indexed fields.
search-service scope
Free-text + fuzzy — "romantic cliffside" → matches title, tagline, description, amenities. Geo radius, Redis caching, cross-field scoring. Cannot be replaced by a simple MongoDB filter.

5 — Notification triggers

Who is responsible for sending emails on booking confirmed, booking cancelled, review approved?

graph LR BK["booking-service"] -->|"POST /api/v1/notifications/send\n{ type: booking_confirmed, user_id, data }"| NT["notification-service"] RV["review-service"] -->|"POST /api/v1/notifications/send\n{ type: review_approved, user_id, data }"| NT NT -->|SMTP| EMAIL["User Email"] style NT fill:#991B1B,color:white style BK fill:#1A3A5C,color:white style RV fill:#1A3A5C,color:white
❌ Rejected approach
Each service implements its own email sending. Duplicate SMTP config, duplicate templates, no central audit trail.
✅ Agreed resolution
notification-service is the only service with SMTP config. Other services fire-and-forget to POST /notifications/send. Internal endpoint — not exposed via gateway to clients.

6 — User registration (Keycloak + PostgreSQL)

Registration must create a Keycloak user AND a PostgreSQL user record atomically. Which service owns this?

sequenceDiagram participant FE as Frontend participant US as user-service :8004 participant KC as Keycloak participant PG as PostgreSQL FE->>US: POST /api/v1/auth/register { name, email, password } US->>KC: Admin API — create user in experience realm KC-->>US: 201 { keycloak_id: "uuid" } US->>PG: INSERT INTO users (id=keycloak_id, email, name...) PG-->>US: user row US-->>FE: 201 { user created — please login via Keycloak }
❌ Rejected approach
Separate register call to Keycloak from frontend + separate API call to user-service. Race condition risk — Keycloak user created but PostgreSQL record fails.
✅ Agreed resolution
user-service owns registration. Creates Keycloak user via Admin API first, then writes PostgreSQL record. Uses Keycloak UUID as the shared key. Login always goes directly to Keycloak token endpoint.

Full Booking Journey — cross-service flow

The most complex user journey: browsing → booking → payment → confirmation email.

sequenceDiagram participant FE as Frontend participant GW as Gateway participant ES as experience-service participant BK as booking-service participant NT as notification-service FE->>GW: GET /api/v1/experiences/abc GW->>ES: forward (public) ES-->>FE: experience detail { title, price, images } FE->>GW: GET /api/v1/experiences/abc/availability?month=2026-07 GW->>BK: forward (public — no token needed) BK-->>FE: { available_dates: ["2026-07-10", "2026-07-11"...] } FE->>GW: POST /api/v1/bookings (with JWT) GW->>GW: forwardAuth → auth-validator → 200 X-User-ID GW->>BK: forward + X-User-ID header BK->>BK: check availability slot BK->>BK: reserve slot (availability.slots_booked++) BK->>BK: create booking (status=pending) BK-->>FE: 201 { booking_id, status: pending } FE->>GW: POST /api/v1/payments/initiate (with JWT) GW->>BK: forward BK-->>FE: { payment_url: "https://stripe/..." } Note over FE: User completes payment on Stripe FE->>GW: POST /api/v1/payments/confirm (with JWT) GW->>BK: forward BK->>BK: UPDATE booking status=confirmed BK->>NT: POST /notifications/send { type: booking_confirmed } NT->>NT: send confirmation email BK-->>FE: 200 { status: confirmed }
